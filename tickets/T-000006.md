## Goal
Confirm that speaking while the assistant is responding (barge-in) immediately cancels the current LLM + TTS output and starts a fresh turn, with no audio artifacts or pipeline hangs.

## Acceptance Criteria
- [ ] Barge-in scenario: assistant speaking → user interrupts mid-sentence → assistant stops within ≤500ms → new response begins
- [ ] No zombie `httpx` streaming tasks remain after interruption (verify with `asyncio` task inspection or logs)
- [ ] `CancelFrame` propagates through `QwenLLMService` and `DeepgramTTSService` without raising unhandled exceptions
- [ ] Rapid successive interruptions (3 in 5 seconds) do not cause pipeline hang or memory leak
- [ ] Clean shutdown: `Ctrl-C` on `voice_pipeline.py` terminates gracefully (no `asyncio` errors in stderr)

## Notes
- Barge-in in Pipecat works via `UserSpeakingFrame` triggering `InterruptionFrame` → `CancelFrame` propagation when `allow_interruptions=True`
- The DeepgramSTTService emits `UserSpeakingFrame` as soon as VAD detects voice activity — confirm VAD is configured in STT service
- Test script: use a loopback audio source or `sox` to feed pre-recorded audio into the WebSocket
- Difficulty: MEDIUM — most bugs here are race conditions in cleanup paths
