## Goal
Assemble `server/voice_pipeline.py` — a runnable Pipecat pipeline connecting WebSocket transport → Deepgram STT → `QwenLLMService` → TTS → audio output, with `allow_interruptions=True`.

## Acceptance Criteria
- [ ] File `server/voice_pipeline.py` is the single entry point for the voice assistant
- [ ] Pipeline order: `transport.input() → DeepgramSTTService → QwenLLMService → DeepgramTTSService → transport.output()`
- [ ] `PipelineParams(allow_interruptions=True)` is set so barge-in propagates `CancelFrame` correctly
- [ ] Server starts on `PIPECAT_PORT` (default 8788) and accepts WebSocket audio from the browser
- [ ] Speaking → Qwen responds → user speaks mid-response → old response stops, new one starts (manual barge-in test passes)

## Notes
- This replaces `server/pipecat_harness.py` as the voice assistant entry point; the harness becomes the STT-only translation tool
- Use `WebsocketServerTransport` from `pipecat.transports.network.websocket_server` — already understood from the harness work
- Conversation context: maintain a `messages` list and pass it as `LLMMessagesFrame` to preserve turn history
- Deepgram STT emits `TranscriptionFrame` (final) — the pipeline needs a small context aggregator to collect user speech before sending to LLM
- Difficulty: MEDIUM — wiring is straightforward once T-000002/003/004 are done; risk is in context management
